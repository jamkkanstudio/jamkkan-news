# Sprint 13 — 오늘의 브리핑 선정 기준 정리

## 상태

구현 완료 · 배포 검증 대기

## 목표와 범위

오늘의 TOP5와 나의 TOP5를 KST 당일 발행 기사 후보군으로 제한하고, 실제 광역 RSS 카테고리, 설명 가능한 동일 사건 제거, 영향·시급성·새로움·분야 다양성, 최대 5개 관점 전환과 안정적인 완료 경험을 적용합니다. 익명 공개 읽기, legacy 성장 흐름, `ADMIN_EMAILS` 페이지·서비스 경계, 자동 수집, JSON·Supabase 동기화, 사용자별 격리와 기존 행은 보존합니다. 유료 API, AI 세부 주제 분류, 기존 데이터 재작성·소유권 변경, 대규모 프런트엔드 이전은 제외합니다.

## 감사 결과와 결정

- 시작 시점 뉴스 56건은 `published_at` 없이 `created_at`만 있었고 모두 `기타`였습니다. 홈은 날짜 필터 없이 전체 이력에 금융·기술 키워드 가산점을 적용해 두 TOP5를 연속 노출했고, 완료 집합도 수집 때마다 바뀔 수 있었습니다.
- 기사 시각은 `published_at`, `collected_at`, legacy `created_at` 순으로 유효한 값을 해석합니다. 기존 Supabase 뉴스 스키마에 새 시각 컬럼을 바로 추가하면 JSON·Supabase 병행 저장이 깨질 수 있어, 신규 저장도 유효 발행·수집 시각을 기존 `created_at` 호환 필드에 보존하고 기존 행은 수정하지 않습니다.
- 자동 수집은 `최신 → 기타` 변환 대신 정치·경제·사회·국제 피드를 통합합니다. 기사 ID·URL 중복은 먼저 제거하고, 한 피드 실패 시 나머지를 사용하되 전부 실패하면 안전한 수집 오류로 처리합니다.
- 동일 사건은 제목의 두 개 이상 핵심어가 높은 비율로 겹치는지 비교해 가장 높은 점수의 대표 기사만 남깁니다. importance 편집값, 사회적 영향 신호, 시급성 신호, KST 최신성을 합산하고 같은 광역 카테고리 반복에는 완만한 감점을 적용합니다.
- KST 당일 후보가 5개 모이면 즉시, 그렇지 않으면 오전 8시 이후 첫 후보를 최대 50개 ID로 고정합니다. 스냅샷은 자동화 `ADMIN_EMAILS` 검사를 다시 통과한 뒤 Supabase `settings`에 먼저 저장하고 `data/settings.json`을 원자 교체합니다. 같은 날짜의 스냅샷은 다시 쓰지 않으므로 두 관점과 완료 대상이 수집마다 흔들리지 않습니다.
- 홈은 `모두에게 중요 | 나에게 중요` 중 한 관점만 렌더링합니다. 두 알고리즘은 같은 고정 후보군에서 각각 최대 5개를 선택하고, 어느 한 관점의 브리핑을 모두 읽으면 `오늘은 충분합니다. 내일 다시 잠깐.`을 보여줍니다. 당일 기사가 5개보다 적어도 오래된 기사는 채우지 않습니다.
- 저장 계약은 광역 카테고리 하나입니다. 세부 `topics`는 최대 2개의 선택적 배열로 설계하되, 기존 Supabase 컬럼과 관리자·분석 호환을 위한 별도 스키마·백필·롤백 결정 전에는 저장하지 않습니다.

## 구현 결과

- KST 당일 필터, 안전한 시각 fallback, 비변경 점수 계산, 사건 중복 제거, 분야 다양성, 개인 관심 가산을 순수 서비스 함수로 분리했습니다.
- 자동·수동 RSS 복구가 실제 광역 분류와 발행 시각을 보존하도록 바꾸고 일일 후보 스냅샷을 JSON·Supabase에 병행 저장하도록 했습니다.
- 홈의 두 개 TOP5 섹션을 한 개 전환형 브리핑으로 줄이고, 당일 기사 부족·후보 고정 중 상태와 완료 문구를 명시했습니다.
