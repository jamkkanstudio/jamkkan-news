# Sprint 11 — 자동 수집 운영 관찰 및 v0.2 마감

## 상태

구현 완료 · 배포 검증 대기

## 목표와 범위

Supabase Cron 자동 수집의 초기 운영 기준선을 만들고 관리자 관측을 최소 보완한 뒤 v0.2를 마감합니다. 익명 공개 읽기, `ADMIN_EMAILS` 관리자 경계, JSON·Supabase 뉴스 동기화, 사용자별 저장은 유지합니다. 다중 피드·AI·유료 API, 기존 뉴스나 사용자 데이터의 삭제·재작성·소유권 변경은 제외합니다.

## 위험과 결정

- GitHub Actions 전체 5회 중 수동 2회와 Supabase Cron 예약 3회가 모두 성공했습니다. 예약 실행 `29397386294`, `29397937214`, `29398501019`는 예정 시각보다 각각 약 1초 뒤 시작해 42초, 41초, 36초에 끝났고 실패는 0회였습니다.
- 세 예약 실행은 각각 기사 5건, 5건, 1건과 데이터 커밋 `10f2046`, `79986a1`, `6b6ac17`을 만들었습니다. 관찰 구간의 예약 실행 3/3이 재배포로 이어졌지만 증가량은 줄기 시작했으므로 빈도는 계속 관찰하되 짧은 표본만으로 수집 주기를 바꾸지 않습니다. GitHub native `schedule` 이벤트가 생성되지 않는 제약도 그대로입니다.
- Sprint 시작 커밋 `5af06c0` 이후 자동 데이터 커밋 두 건으로 뉴스는 18건에서 24건이 됐습니다. 현재 JSON은 16,527바이트이고 기사 ID와 URL은 각각 24개 모두 고유합니다.
- 보존 정책은 아직 적용하지 않습니다. `data/news.json`이 5MiB, 기사 5,000건, 또는 하루 데이터 커밋 24회를 7일 연속 넘는 첫 시점에 별도 Sprint를 엽니다. 후보 정책은 안정 ID를 유지한 월별 불변 보관본과 최근 활성 창을 분리하는 방식입니다. 적용 전 JSON과 Supabase의 ID·URL·행 수 스냅샷, Git 복구 지점, 사용자 성장·분석 참조 불변 검증을 만들고, 실패 시 이전 JSON 커밋 복원과 보관 행 재-upsert가 가능해야 합니다.
- 관리자 화면은 성공 시각과 건수는 설명했지만 실패 시도 시각과 직전 성공, 오래된 성공 상태를 구분하지 못했습니다. 상태 저장 형식은 바꾸지 않고 KST 시각, 실패 시도·직전 성공, 마지막 성공 후 30분 지연 경고만 추가합니다. 원문 예외·URL·비밀값은 계속 표시하지 않습니다.
- Actions 전용 PAT는 `jamkkan-news`의 Actions 읽기·쓰기만 허용하고 2026-10-13 만료입니다. 2026-09-15에 같은 최소 권한의 새 토큰으로 Vault UI에서 교체하고 다음 예약 성공을 확인한 뒤 이전 토큰을 폐기합니다. 실패하면 아직 유효한 이전 토큰으로 Vault 항목을 되돌립니다. 비밀값은 코드·Git·SQL history·로그·채팅에 입력하지 않습니다.

## 구현 결과

- 관리자 자동 수집 상태의 시각을 KST로 통일했습니다.
- 마지막 성공이 30분 이상 오래되면 정상 대신 지연 경고를 표시합니다.
- 실패 상태에 실패 시도 시각과 직전 성공 시각을 함께 표시하면서 안전한 오류 코드만 유지합니다.
- v0.2 마감 결과와 운영용 PAT 교체 체크포인트를 문서화했습니다. `ROADMAP.md`의 최종 상태는 배포 검증 뒤 변경합니다.
